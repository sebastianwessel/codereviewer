# 03: Finding, Evidence, And Report Contracts

Status: Approved
Date: 2026-06-19

## Contract Source Rule

Implementation must define Zod schemas for every contract in this file under
`src/shared/contracts/`. TypeScript types must be inferred from schemas. Tests
must validate every fixture against schemas.

## Path Contract

- `repositoryRelativePath`: portable path relative to repository root.
- It must use `/` separators.
- It must not be empty, absolute, contain NUL, or traverse above root.
- It must preserve case as found on disk.
- Comparisons must use normalized comparison keys generated by `path-service`.

## Location Contract

`CodeLocation` fields:

| Field | Required | Type | Semantics |
| --- | --- | --- | --- |
| `path` | yes | repositoryRelativePath | File path in repository. |
| `startLine` | yes | integer >= 1 | 1-based line in head file unless `side` is `old`. |
| `startColumn` | no | integer >= 1 | 1-based UTF-16 column. |
| `endLine` | no | integer >= startLine | Inclusive line end. |
| `endColumn` | no | integer >= 1 | Inclusive UTF-16 column end. |

Admission must reject candidates whose `CodeLocation` cannot be resolved to a
reviewed file.

## Related Location And Data Flow Contracts

`RelatedLocation` records supporting repository locations that are not the
primary finding location.

| Field | Required | Type | Rule |
| --- | --- | --- | --- |
| `id` | yes | string | Stable within the finding. |
| `location` | yes | `CodeLocation` | Must resolve to repository content reviewed or indexed in the run. |
| `message` | yes | string 1..300 | Redacted, safe for report. |

`DataFlowPath` records ordered source-to-sink or cause-to-effect locations.

| Field | Required | Type | Rule |
| --- | --- | --- | --- |
| `id` | yes | string | Stable within the finding. |
| `label` | yes | string 1..120 | Example: `tainted-input-to-sink`. |
| `steps` | yes | non-empty `RelatedLocation[]` | Ordered; every step must resolve. |

## Fingerprint Contract

`FindingFingerprint` fields:

| Field | Required | Type | Rule |
| --- | --- | --- | --- |
| `algorithm` | yes | string | Stable algorithm identifier, for example `v1-rule-path-location-title`. |
| `value` | yes | string | Lowercase base32/hex hash. |

R1 must generate at least one fingerprint per admitted finding from normalized
category, rule ID when present, repository-relative path, normalized location,
title, and evidence kind. Fingerprints drive de-duplication and baseline status;
they must not include raw source text, absolute paths, provider response IDs, or
timestamps.

`report.json` is the canonical machine-readable artifact and is always written.

Markdown and SARIF. The artifact writer may separately return metadata for the
JSON file to callers, but that JSON self-entry is not embedded in `report.json`.

## Markdown Report

Markdown rendering must be deterministic from `ReviewReport`:

1. Title and run summary.
2. Quality gate result if present.
3. Counts by severity and category.
4. Admitted findings sorted by severity, path, line, title.
5. Rejected candidate summary by reason.
6. Skipped files.
7. Cost/timing summary.

Markdown must not include raw source snippets by default.

## SARIF Report

R1 must render `report.sarif` from the canonical `ReviewReport`. SARIF is an
export format, not the internal domain model.

Rules:

- emit SARIF `2.1.0`;
- use repository-relative URIs with `/` separators;
- never embed raw source text, command lines, environment variables, local
  absolute paths, user names, or machine names;
- map one internal finding to one SARIF result unless a future spec defines
  aggregation;
- map related locations and data flow into SARIF locations/code flows only
  after each path validates through `path-service`;
- include stable partial fingerprints derived from `FindingFingerprint`;
- set a deterministic `runAutomationDetails.id` when a configured SARIF
  category exists;
- cap rendered results through `reporting.sarif.maxResults`;
- define every referenced `ruleId` in `tool.driver.rules`;
- validate the rendered file before writing it.

Generic structural validation (always enforced):

- `version` is `"2.1.0"` and `runs` is a non-empty array;
- `runs[0].tool.driver.name` is a non-empty string;
- each result has a non-empty `ruleId`, a string `message.text`, and at least one
  location whose `artifactLocation.uri` is a repository-relative URI reference
  (no leading `/`, drive letter, `..`, or backslash) and whose `region.startLine`
  is an integer `>= 1`.

GitHub-supported subset (enforced additionally when `reporting.sarif.target` is
`github`):

- every result has non-empty `partialFingerprints` for stable de-duplication;
- every result's `ruleId` is defined in `tool.driver.rules`;
- the number of distinct rules does not exceed `1000`.

Validation failures are reported as a deterministic report error (exit code `5`)
and the artifact is not written.

SARIF upload, code scanning alert management, and PR annotation publication are
out of scope for R1. R1 only writes the local artifact.

## Compatibility

`schemaVersion` changes:

- patch-compatible additions must be optional;
- removing or changing required fields requires `schemaVersion` increment;
- report readers must reject unknown major versions.
